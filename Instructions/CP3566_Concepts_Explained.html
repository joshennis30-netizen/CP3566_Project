Understanding the Project:
The Ideas Behind It

Before you write any code, read this. The fraud project uses a few ideas you have not used yet — web servers, Application Programming Interfaces (APIs), layers, logins. This guide explains each one in plain words, with a picture. It follows the same order you will build things in.

01The big picture

Your whole app is three layers of code sitting between a user and a database. A user (or another program) sends a request in; the app does some work; an answer goes back out. Everything you build fits somewhere in the picture below.

02How the web works

Every web app is two halves talking to each other: a client (the browser) and a server (your app, running on a computer somewhere). They talk in a strict back-and-forth.

The client sends a request — basically “please do this”. The server sends back a response — the result, plus a small status number that says whether it worked. The language they speak is called HTTP — short for Hypertext Transfer Protocol. That is all “the web” really is: requests going one way, responses coming back.

📮

Think of posting a letter. You send a request (the letter asks for something). Later you get a reply. You never walk into the post office and sort the mail yourself — you only send and receive. The browser and server work the same way.

03APIs and REST

What is an API?

An API is a fixed list of things one program will let another program ask it to do. Nothing more. It is a doorway with a set menu.

🍽️

Think of a restaurant. You don’t walk into the kitchen and cook. You read the menu (the fixed list of things you can order), tell the waiter, and the food comes back. The menu is the API: it’s the agreed list of requests you’re allowed to make, and what you’ll get back. You don’t need to know how the kitchen works.

In your project, the “kitchen” is your app and its database. The API is the menu of actions you let others use: list the cases, open a case, add a note, close a case. A program (or your own web pages) places an order; your app does the work and sends back the result.

What is a REST API?

REST — short for Representational State Transfer — is just the most common style of web API. It has two simple ideas:

1. Web addresses name the things. Each kind of thing gets an address. In your app, /api/cases means “the cases” and /api/cases/42 means “case number 42.”

2. HTTP methods say what to do with that thing. The method is the verb; the address is the noun. “GET the cases” means read them. “POST a case” means create one.

Method (verb)What it meansExample in your app
GETRead / fetch something. Changes nothing.GET /api/cases — list the cases
POSTCreate something new, or run an action.POST /api/cases/42/notes — add a note

What comes back: JSON

When your API answers, it usually sends the data as JSON — short for JavaScript Object Notation, a simple text format of names and values that any program can read. For one case it might look like { "id": 42, "status": "Escalated", "amount": 12000 }. You do not have to design JSON by hand; your framework turns your Java objects into JSON for you.

The status number

Every response carries a 3-digit status code that says how it went. You must send the right one — the marker checks for this.

CodeMeaningWhen you send it
200OKIt worked (a read or an action succeeded).
201CreatedYou made something new (a note, a transaction).
400Bad requestThe data they sent was broken or missing.
401Not logged inThey have not logged in yet.
403Not allowedThey are logged in, but their role can’t do this.
404Not foundThe thing they asked for doesn’t exist.
409ConflictThey tried a move that isn’t allowed (e.g. an illegal case step).
In your projectSection 4.5 of the spec lists the exact addresses and methods you must build. Now you can read that table: POST /api/cases/{id}/escalate means “run the escalate action on this one case.”

04The layers of your app

A good app is not one big pile of code. It is split into layers, each with one job. A request travels down through the layers and the answer travels back up. This keeps the “thinking” separate from the “pages” and from the “database code,” so each part stays simple.

🏛️

You may hear the name “MVC.” It is the same idea with three words: the View is what the user sees (the pages), the Controller takes the request and decides what to do, and the Model is the data. Keeping them apart is exactly what the layers above do.

05The database and JPA

Your app forgets everything the moment it stops running — unless it saves things in a database. The database is where your transactions, cases, notes, and log lines actually live.

Instead of writing SQL by hand, this project talks to the database through JPA — the Jakarta Persistence API — using Spring Data. You mark a class @Entity and declare a repository interface; the framework (Hibernate) writes the SQL and turns rows into objects for you. You met databases in the prerequisite course; JPA is the modern way Java reaches them — under the hood it still uses JDBC, but you don’t write that plumbing.

Parameterised queries — safe by default

When a query uses a value that came from outside (a username, an amount), that value must never be glued straight into the text of the query. With JPA you get this for free: the repository methods (and any @Query you write with bound parameters) send the value separately from the query — a parameterised query.

🛡️

Why it matters. If user input is pasted into your query text, a clever user can type something that changes what the query does — this attack is called SQL injection (SQL is short for Structured Query Language), and it can hand over your whole database. Letting JPA bind parameters closes that door. The rule: use the repositories, and if you ever write a custom query, bind its parameters — never build SQL by joining strings. The spec requires this; skipping it loses marks.

06Login vs. permissions

Two different checks guard your app, and people mix them up. Logging in answers “who are you?” Permissions answer “what are you allowed to do?” You need both, and they happen in that order.

The proper names are authentication (who you are — the login) and authorisation (what your role may do). An analyst is authenticated when they log in, but they are not authorised to confirm fraud.

One rule about the login itself: never store passwords as plain text. Store a one-way hash of each password; when someone signs in, hash what they typed and compare it to the stored hash. The starter already does this for you (with the standard BCrypt function), so a wrong password is rejected with 401 — the same code as “not logged in.”

07The rule engine

This is the heart of the project. When a transaction is saved, your app runs it past a short list of rules. Each rule is a plain check. If any rule matches, the transaction is suspicious: the app raises an alert and opens a case for a person to look at.

The rules are simple on purpose — is the amount too big, are there too many transactions too fast, is the name on the watchlist. There is no “smart” guessing involved.

In your projectThe rule numbers (the dollar limit, the time window) live in the database so an admin can change them. Do not hard-code them. The optional R4 “splitting up” rule earns bonus marks.

08The life of a case

A case is never just “open” or “closed.” It moves through a fixed set of stages, and only certain moves are allowed. Your app must refuse any move that isn’t on this map and answer with an error (status 409) instead of allowing it.

🚦

This is called a “state machine.” The case is always in exactly one state, and the arrows are the only legal exits. Your code’s job is to check, before every move, that the arrow exists — and that the person’s role is allowed to take it.

10What’s given, what to build

Everything above is the idea. Here is the deal: a few pieces are handed to you so you don’t waste time on plumbing, and the rest is yours to write. Build it in the order of the layers from §4 — starting from the data layer you were given, then logic, then the web.

11Quick glossary

One line each, for when you forget a word halfway through building.

Client
The program asking for something — here, the web browser.
Server
The program answering — your app, running on a computer.
HTTP
The set back-and-forth language the web uses: a request out, a response back.
Request / Response
“Please do this” / “Here is the result, plus a status number.”
API
The fixed list of actions one program lets another ask it to do.
REST API
A web API where addresses name the things and HTTP methods (GET, POST, PUT, DELETE) say what to do.
JSON
A simple text format of names and values used to send data back.
Status code
A 3-digit number on every response: 200 worked, 401 not logged in, 403 not allowed, 409 illegal move.
Layer / tier
A part of the app with one job: pages, logic, or database code.
MVC
Model (data), View (pages), Controller (handles requests) — the layers, by another name.
ORM
Object–Relational Mapping: lining up a class with a table, a field with a column, an object with a row.
JPA / Spring Data JPA
The standard (and Spring’s helper) for storing objects in a database without writing SQL by hand.
Entity
A class marked @Entity that maps to a database table.
Repository
An interface that gives you database operations (save, find, delete) without an implementation you write.
Hibernate
The engine under JPA that actually writes the SQL and maps rows to objects.
Parameterised query
A query with its values passed in separately — safe from SQL injection; JPA does this for you.
Authentication
Checking who you are — the login.
Authorisation
Checking what your role may do — the permissions.
Rule engine
The simple checks that decide if a transaction is suspicious.
Alert
What a matched rule raises; it opens a case.
State machine
A thing that is always in one state, with only certain allowed moves between states — your case workflow.
Audit log
The permanent, unchangeable record of every action.
